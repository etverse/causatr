# causatr

Unified causal effect estimation for methodological triangulation: g-computation
(parametric g-formula + ICE), IPW (self-contained density-ratio engine), and matching (via MatchIt).
Part of the [etverse](https://github.com/etverse) ecosystem.

## S3 classes

- `causatr_fit` — returned by `causat()`. Holds model(s), data, method, and metadata.
- `causatr_result` — returned by `contrast()`. Holds estimates, SEs, CIs, vcov.
- `causatr_diag` — returned by `diagnose()`. Holds balance, positivity, weight info.
- `causatr_intervention` — returned by `static()`, `shift()`, etc.

## Two-step API

```r
# Step 1: fit
fit <- causat(data, outcome = "Y", treatment = "A", confounders = ~ L1 + L2,
              estimator = "gcomp",        # or "ipw" / "matching"
              model_fn = stats::glm)   # or mgcv::gam, etc.

# Step 2: contrast
result <- contrast(fit,
  interventions = list(a1 = static(1), a0 = static(0)),
  type = "difference",            # or "ratio" / "or"
  ci_method = "sandwich")         # or "bootstrap"
```

## Scope

causatr is a **causal effect estimation package** for methodological triangulation. It owns outcome-model-based g-computation (parametric g-formula + ICE), a self-contained IPW density-ratio engine, and delegates matching to MatchIt. It is NOT: a TMLE/debiased-ML package (use `lmtp`), a mediation package, a sensitivity analysis package, a heterogeneous treatment effects package (use `grf`), or a forward-simulation g-formula (use `gfoRmula`).

### Why ICE, not forward simulation

The book (Hernán & Robins, Ch. 21) describes two longitudinal g-computation approaches. **Forward simulation** (gfoRmula-style) requires models for every time-varying covariate + the outcome, simulates trajectories forward, and needs bootstrap-only inference (slow, p+1 models to get right). **ICE** (Zivich et al. 2024, Stat in Med 43:5562-5572) models the outcome only at each time point, iterates backward using pseudo-outcomes, and admits stacked estimating equations with empirical sandwich variance. Far fewer models, far faster. For point treatments, ICE reduces to standard g-computation / standardization (Ch. 13).

### Why GLMs/GAMs, not ML

Black-box ML in g-computation is not automatically valid for causal inference: plug-in g-computation with ML can be $\sqrt{n}$-inconsistent due to overfitting bias unless combined with debiasing (cross-fitting, TMLE). causatr supports flexible-but-parametric models (GLMs with splines, GAMs) where the sandwich variance estimator is valid. Users who want ML-based estimation should use `lmtp`.

## Key design decisions

- **ICE over forward simulation** — fewer models, sandwich variance, faster
- **GLMs/GAMs over ML** — valid sandwich inference without debiasing
- **Three methods** — gcomp (outcome model), IPW (treatment model), matching. Agreement $\to$ confidence.
- **MatchIt is delegated; WeightIt is a contrast-level test oracle.** Matching delegates to `MatchIt` because building it correctly is genuinely hard. IPW is owned end-to-end via a single self-contained density-ratio engine that handles static + shift + scale_by + dynamic (binary) + IPSI uniformly using the user's `model_fn` / `propensity_model_fn` for the treatment density model. `WeightIt` is invoked only by tests as an independent reference for the static binary case and is not on the runtime path.
- **`model_fn` parameter** — user passes the fitting function (glm, gam, etc.) rather than hardcoding
- **`estimator`, not `method`** — `causat()` / `causat_survival()` take `estimator = c("gcomp", "ipw", "matching")`. The name is deliberately NOT `method` so `...` can forward cleanly to `MatchIt::matchit(method = ...)` without a shadowing collision (e.g. `causat(data, ..., estimator = "matching", method = "nearest")` "just works"). When adding a new wrapper around another package, preserve this invariant: never use `method =` as a top-level causat argument.
- **Sandwich default** — fast, valid for GLMs; bootstrap fallback for GAMs
- **Intervention functions, not formulas** — maximum flexibility for MTPs, dynamic rules, IPSI
- **`contrast()` as separate step** — fit once, contrast many interventions
- **ci_method: sandwich or bootstrap only** — delta method is applied internally for ratio/OR contrasts; not a separate ci_method
- **Single influence-function engine** — `variance_if()` (`R/variance_if.R`) is the one entry point for sandwich variance across g-comp, IPW, matching, and ICE. All four methods compute a per-individual IF as Channel 1 (direct covariate-sampling term) + Channel 2 (nuisance-model correction) and aggregate via `vcov_from_if()`. G-comp and matching go through `prepare_model_if()` / `apply_model_correction()` (one $p \times p$ bread solve per model, reused across interventions); matching adds `cluster = subclass` for within-subclass aggregation. IPW goes through `compute_ipw_if_self_contained_one(fit, intervention)` per intervention — Channel 1 is the propensity score, Channel 2 is the cross-derivative `numDeriv::jacobian` of `make_weight_fn()` propagated by `sandwich::estfun` + `sandwich::bread` on the propensity model, and the result is the per-individual IF on the stacked `(alpha, beta)` M-estimation system. ICE iterates `correct_model()` once per model in a forward sensitivity recursion. See `vignettes/variance-theory.qmd` Sections 3–4 for the derivation and the `R/variance_if.R` roxygen header for the engine's internal layout.
- **`type` parameter** — `causat()` accepts `type = NULL` (default, auto-detected from `id`/`time`), `"point"`, or `"longitudinal"`. Explicit `type = "longitudinal"` requires `id` and `time`.
- **Unified self-contained IPW** — `fit_ipw()` builds a treatment density model via `fit_treatment_model()` using the user's `propensity_model_fn` (defaults to `model_fn`). `compute_contrast()` then refits a per-intervention weighted MSM `Y ~ 1` on density-ratio weights from `make_weight_fn()`, recovering each counterfactual marginal mean as the Hájek intercept. The same path covers static (HT indicator), shift / scale_by (continuous pushforward + Jacobian), dynamic on binary (HT indicator on a deterministic rule), IPSI (closed-form $w_i = (\delta \cdot A_i + (1 - A_i)) / (\delta \cdot p_i + (1 - p_i))$), and count treatments (Poisson / NB via `propensity_family = "poisson"` / `"negbin"`, integer shift / scale only). **Estimand gating:** ATT / ATC are only defined for static interventions on a binary 0/1 treatment — `check_estimand_intervention_compat()` aborts any other combination at contrast time with error class `causatr_bad_estimand_intervention`. For the cases that *are* allowed, the HT weight branch carries a Bayes-rule numerator $f^*_i = f(A^* \mid L_i)$ so the same density-ratio formula produces ATE, ATT, and ATC weights uniformly: $w_i = \mathbf{1}\{A_i = a\} \cdot f^*_i / f(a \mid L_i)$, with $f^* = 1$ (ATE), $f^* = p(L)$ (ATT), $f^* = 1 - p(L)$ (ATC). The closure inside `make_weight_fn()` bakes the same $f^*(p(\alpha))$ into the $\alpha$-dependent weight so the variance engine's `numDeriv::jacobian(weight_fn)` stays consistent; the ATT `a=1` and ATC `a=0` arms correctly reduce to $\alpha$-independent weights, matching the fact that those direct-sample functionals carry no propensity-score uncertainty. Chunks 3c.i–3c.iii wired the density-ratio runtime but did not consume `estimand`; chunk 3d both added the `WeightIt::glm_weightit()` contrast-level oracle (T-oracle1..3) and fixed this latent ATT/ATC bug — the sandwich SEs now agree with `glm_weightit` to ~1e-6 on the same propensity model. **Rejected combinations:** `static()` / `threshold()` / `dynamic()` on continuous treatment, and any `A:modifier` confounder term, all hard-abort upfront with a flat factual error pointing at `estimator = "gcomp"`. **Oracles:** `WeightIt` is no longer on the runtime path and serves only as a contrast-level test oracle on the static binary case (`tests/testthat/helper-ipw-weightit-oracle.R` + `test-ipw-weightit-oracle.R`); `lmtp::lmtp_ipw()` is the point-estimate oracle for shift / IPSI (its EIF-based SE is not directly comparable to the M-estimation sandwich, so it is not used as a variance oracle).

## Architecture notes

- The NHEFS dataset ships with 1746 rows (117 have NA education). The book uses a 1629-row pre-cleaned subset. `compute_contrast()` handles this by excluding rows with NA predictions from the target population.
- **Variance engine architecture (`R/variance_if.R`).** One dispatcher `variance_if(fit, ...)` routes to four method-specific branches that share a small set of primitives: `bread_inv()` ((X'WX)^-1 for GLMs with a `rlang::warn()` on the `MASS::ginv()` fallback, `$Vp` for GAMs); `iv_design_matrix()` (counterfactual model matrix, GLM/GAM split); `prepare_model_if()` / `apply_model_correction()` (Channel 2 per model, split into a gradient-independent prep step and a per-intervention apply step so a single `p x p` bread solve serves all interventions); `correct_model()` (thin wrapper for single-gradient callers, kept for ICE); `compute_ipw_if_self_contained_one()` (per-intervention IPW IF on the stacked `(alpha, beta)` M-estimation system — Channel 1 from the propensity score, Channel 2 from the cross-derivative `numDeriv::jacobian` of the weight closure, propagated via `sandwich::estfun` + `sandwich::bread` on the propensity model); `resolve_fit_idx()` (asserts the `na.action` invariant for the model's fit-row mapping); `vcov_from_if()` (aggregates IFs to a $k \times k$ vcov via `crossprod`, with optional `cluster =` for matching's within-subclass sum-then-square); `variance_if_numeric()` (two-tier fallback — Tier 1 recovers the full IF via `sandwich::estfun + bread`, Tier 2 falls back to $V_1 + J V_\beta J^T$ where V1 is the full `crossprod(Ch1_mat) / n^2` so off-diagonal Channel-1 covariance survives, with a warning about the dropped IF cross-term).
- **G-comp / matching** prepare once per model (`prepare_model_if`) before the intervention `lapply`, then reuse the prep across interventions — O(1) bread solves instead of O(k). **Matching** works entirely on `fit$details$matched_data` (n = n_match, cluster-robust on `subclass`) per the IF scope from vignette Section 4.3. **ICE** iterates `correct_model()` once per model in a forward sensitivity recursion, consuming `$d` from step k to build step k+1's gradient; ICE's outer loop cycles models rather than interventions so prep hoisting does not apply. **IPW** runs `variance_if_ipw()` as a straight `lapply` over interventions calling `compute_ipw_if_self_contained_one(fit, intervention)`, which builds the per-individual IF on the stacked `(alpha, beta)` M-estimation system and aggregates via `vcov_from_if()`. The Phase-3-era `glm_weightit` / Mparts machinery is gone; `WeightIt` is no longer on the runtime path.
- **`dynamic()` vs MTPs — terminology gotcha.** The literature uses "dynamic regime" loosely for two distinct objects: (1) deterministic per-individual rules `d(L_i)` that pick out a single counterfactual treatment value (Robins; Murphy; Hernán & Robins Ch. 21), and (2) **modified treatment policies** — smooth / stochastic transformations of the observed exposure like $A \to A + \delta$, $A \to \delta \cdot A$, or incremental propensity-score shifts (Díaz & van der Laan; Haneuse & Rotnitzky; Kennedy 2019). causatr's `dynamic()` constructor is reserved for sense (1), which is why it rejects continuous treatment under IPW (a deterministic rule on a continuous A is a Dirac per individual with no Lebesgue density). MTPs in sense (2) are fully supported under IPW on continuous treatment via `shift()`, `scale_by()`, and `ipsi()` — those go through the smooth pushforward branch in `R/ipw_weights.R` with the correct Jacobian and serve the same scientific use cases the literature's "continuous dynamic regimes" usually mean.
- **IPW MSM is `Y ~ 1` (intercept-only Hajek) by default, not `Y ~ A`.** Under the unified density-ratio engine the per-intervention weight collapses every nonzero row onto the same counterfactual treatment value (the HT indicator path) or onto a smooth pushforward over the same observed `A`, so a saturated `Y ~ A` would be rank-deficient. `compute_contrast()` refits an intercept-only weighted GLM per intervention and reads `E[Y^d]` off the (sole) coefficient via the standard predict-then-average path. **With effect modification** (e.g. `A:sex` in confounders), the MSM expands to `Y ~ 1 + modifier` via `build_ipw_msm_formula()` so that `predict()` returns stratum-specific counterfactual means; the treatment is still absorbed by the density-ratio weights. Matching keeps `Y ~ A` (or `Y ~ A + modifier + A:modifier` with EM) because its weights are subclass indicators rather than HT zeros. Bare treatment in confounders (`~ L + A`) is rejected by `check_em_compat()`.
- The `model_fn` parameter (default `stats::glm`) allows users to plug in any fitting function with signature `function(formula, data, family, weights, ...)`.
- **ICE intervention application — current-time treatment only.** `ice_apply_intervention_long()` applies the intervention to the current treatment column **only** and DOES NOT recompute treatment lag columns from the intervened values. Lag columns hold OBSERVED A_{k-1}, A_{k-2}, ... at every backward step, matching the canonical Robins iterated conditional expectation algorithm and `lmtp::lmtp_tmle()`'s sequential-regression shortcut. A previous implementation recomputed lag columns from the intervened treatment, which double-counted shift/scale/threshold/dynamic interventions: a `shift(delta)` intervention on a 2-period continuous treatment returned 3*delta instead of the structural 2*delta. Verified against lmtp on the linear-Gaussian DGP in `test-simulation.R`.
- ICE g-computation (`R/ice.R`) defers model fitting to `contrast()` because the sequential outcome models are intervention-dependent. `fit_ice()` stores metadata; `ice_iterate()` runs backward iteration per intervention.
- ICE sandwich variance uses stacked estimating equations (Zivich et al. 2024) implemented via `variance_if_ice()` — a forward sensitivity recursion calling `correct_model()` once per model (no `geex` dependency). The $J V_\beta J^\top$ shortcut does NOT work for ICE (ignores upstream model uncertainty, can underestimate SE by 40%+). Note: ICE models are fit *backward* in time ($K \to 0$) inside `ice_iterate()`, but sensitivities propagate *forward* ($0 \to K$) inside the variance loop — both are the back-substitution of the block-triangular bread of the stacked M-estimation system, in different orderings. See `variance-theory.qmd` Section 5.4.
- ICE bootstrap resamples individuals (all person-period rows together) via `boot::boot()` with `parallel`/`ncpus` support. `contrast()` passes these through.
- For binary outcomes, the first ICE model uses `binomial` (actual 0/1 response), but pseudo-outcome models use `quasibinomial` (fractional logistic for predicted probabilities in [0,1]).
- Matching with `estimand = "ATE"` auto-selects `method = "full"` for MatchIt (nearest-neighbor only supports ATT/ATC). User can override via `...`.
- Bootstrap wraps the entire replicate in `tryCatch` — failed replicates (e.g. factor level mismatches in bootstrap samples) return NA and are excluded from the variance estimate (standard bootstrap practice; Davison & Hinkley 1997). A warning is issued when any replicates fail.
- The `by` parameter in `contrast()` stratifies estimates by levels of a variable. Internally, it loops over levels, constructing a subset expression for each and delegating to the standard contrast machinery. Results include a `by` column in the estimates and contrasts data.tables. **`by` is a stratification mechanism, not a refit.** It averages the already-fitted model's predictions over each subset. For genuine *effect modification*, the underlying model must include an interaction term (e.g. `confounders = ~ L + sex + A:sex` for gcomp). Today only point gcomp recovers heterogeneous effects correctly under `by`; IPW / matching / ICE have partial or missing interaction support (see Phase 8).
- `causatr_result` stores `family` and `fit_type` from the fit, enabling context-aware printing, plotting (e.g. "risk difference" vs "mean difference"), and log-scale choice for ratio/OR contrasts.
- `plot.causatr_result()` uses the `forrest` package (Suggests) for publication-ready forest plots. Adapts reference line, log scale, and axis labels to the contrast type and outcome family.
- `tidy.causatr_result()` and `glance.causatr_result()` provide broom-compatible outputs via `generics` (Imports).
- `knit_print.causatr_result()` (`R/knit_print.R`) renders a `causatr_result` as a metadata header + two tinytable HTML tables (intervention means + contrasts) when the object is emitted from a knitr/Quarto chunk. It is registered in `R/zzz.R`'s `.onLoad()` hook — conditionally, only when `knitr` is installed — so both `knitr` and `tinytable` stay Suggests-only. If either is missing at render time, the method falls back to `knitr::normal_print()`, which in turn dispatches to `print.causatr_result()`. Vignettes use `code-fold: show` in their YAML so readers can fold the code chunk and still see the formatted result table; `format_tt(markdown = TRUE)` (requires `litedown`, also Suggests) is used where table cells carry LaTeX math so `\(...\)` reaches MathJax.
- `confint.causatr_result()` recomputes CIs from the stored SE and vcov, correctly respecting the `level` argument.
- Multivariate treatments (`treatment = c("A1", "A2")`) are supported for g-computation (point and longitudinal). IPW and matching block multivariate treatment with an informative error (Phase 8).
- IPW weight diagnostics (`compute_weight_summary()` in `R/diagnose.R`) reconstruct the propensity score from `fit$details$propensity_model`, build the `treated` / `control` / `overall` HT weights, and compute mean / sd / min / max / ESS per group. The shim covers the binary static ATE case; non-binary treatment families abort with `causatr_diag_unsupported_family`.
- IPW supports binary, continuous, categorical, and count treatments via the unified density-ratio engine (`fit_treatment_model()` + `compute_density_ratio_weights()` + per-intervention `Y ~ 1` Hájek MSM). Categorical treatment uses `nnet::multinom` by default (auto-selected when `propensity_model_fn = NULL` and the treatment is factor/character); the user can override with any multinomial fitter via `propensity_model_fn`. **Count treatments** (Poisson / negative binomial) are enabled via explicit `propensity_family = "poisson"` / `"negbin"` opt-in on `causat()`. Poisson uses `stats::glm(family = poisson())`; NB uses `MASS::glm.nb` (auto-selected when `propensity_model_fn = NULL` and `propensity_family = "negbin"`). Count densities use `dpois()` / `dnbinom()` in the density ratio, with `theta` (NB) treated as fixed in the variance engine (same convention as `sigma` for Gaussian). Only `shift()` (integer delta) and `scale_by()` (inverse must preserve integer support) are allowed; `static`/`dynamic`/`threshold`/`ipsi` are rejected. The variance engine handles Poisson and NB GLMs out-of-the-box via `prepare_model_if()` and `bread_inv()` (both inherit from `glm`). The `make_weight_fn()` closure uses log-link `lambda = exp(X %*% alpha)` for count families. The variance engine handles the multinomial propensity model via `prepare_model_if_multinom()`, which computes bread and score from the multinomial log-likelihood directly (no `sandwich::estfun` method needed). Categorical IPW supports `static()` and `dynamic()` under ATE only (ATT/ATC rejected by `check_estimand_trt_compat()` because factor values don't pass the 0/1 gate; `shift`/`scale_by`/`ipsi`/`threshold` rejected by `check_intervention_family_compat()` because they require numeric structure). Matching is **binary-only**: `MatchIt` itself rejects any non-binary treatment, and `fit_matching()` intercepts upstream with a clear causatr-side error pointing users to `gcomp` / `ipw`.
- **External weights** are validated up front in `causat()` and `causat_survival()` via `check_weights()`: NA, Inf / NaN, negative, non-numeric, and mis-sized vectors are rejected with specific errors. This prevents silent NaN estimates or cryptic downstream GLM aborts. Zero weights are allowed as a pass-through.
- **`na.action = na.exclude` is rejected** in `causat()` / `causat_survival()` via `check_dots_na_action()` (error class `causatr_bad_na_action`). Under `na.exclude`, `residuals(model, "working")` is padded with NAs to the original data length while `model.matrix(model)` drops NA rows; `prepare_model_if()` then builds `r_score` at the wrong length and `apply_model_correction()`'s `d_fit * r_score` silently recycles into a garbage correction vector (R only emits a "longer object length" warning). Only `na.omit` (default) and `na.fail` are accepted. 2026-04-15 fourth-round critical review.
- **`censoring =` is a row filter, not IPCW.** The `censoring` parameter tells `get_fit_rows()` to exclude rows where `C != 0` from model fitting. No censoring model is fit, no IPCW weights are computed. `contrast()` predicts on ALL rows (including censored), so the g-formula standardization is over the full target population. This is the same approach as Zivich et al. (2024): censoring enters the estimating equations as indicator functions `I(C_{i,k} = 0)` restricting each model to uncensored rows. Under censoring exchangeability ($Y^a \perp C \mid A, L$) and a correctly specified outcome model, this recovers the truth without IPCW. Built-in IPCW (fit P(C=0|A,L) internally, compute stabilized weights) is planned for Phase 14 (`PHASE_14_IPCW.md`). Users can already supply manual IPCW weights via `weights =` today.
- **Missing data handling across methods.** Three variables can be missing: outcome (Y), treatment (A), and covariates (L). The handling depends on the variable and the missingness mechanism. **Y missing (MCAR):** dropped via `get_fit_rows()`. Under g-comp, the outcome model is fit on complete cases and predictions are standardized over all rows — unbiased under MCAR. **Y missing (MAR on A, L):** g-comp with correct outcome model is still consistent (the regression surface E[Y|A,L] is unchanged by censoring). IPW requires IPCW weights (Phase 14). Manual IPCW via `weights=` works today. **A missing:** `check_treatment_nas()` aborts — you cannot define a counterfactual without knowing A. User must drop rows (MCAR) or use multiple imputation (MAR, planned via `causat_mice()`). **L missing (gcomp):** `glm(na.action = na.omit)` drops rows with NA covariates. `predict()` returns NA for NA-L rows, which `contrast()` excludes from the target population via `valid_preds`. Under MCAR, unbiased. Under MAR, need MI. **L missing (IPW):** `fit_treatment_model()` drops rows with NA L. If the propensity mask diverges from the outcome mask, `fit_ipw()` aborts — user must drop or impute. **L missing (matching):** MatchIt handles internally via its own `na.omit`. **L missing (ICE):** all-NA columns at a time step are dropped from the formula; partial NAs work under bootstrap but the sandwich cascade gradient has a known alignment issue. **IPW outcome-NA fix (2026-04-16):** `compute_ipw_contrast_point()` and `variance_if_ipw()` now pass the outcome-filtered `fit_data` (not the full `data`) to `compute_density_ratio_weights()` and `make_weight_fn()`, because `tm$fit_rows` from `fit_treatment_model()` is relative to the outcome-filtered subset.
- **ICE sandwich under non-uniform weights (2026-04-14 fix).** `variance_if_ice_one()` builds the step > 0 cascade gradient as `g_k = sum_j w_{k-1,j} d_{k-1,j} X^*_{k,j} mu_eta_{k,j}`. The `w_{k-1,j}` factor comes from `A_{k-1,k} = E[d s_{k-1,j}/d beta_k]`, which carries the prior weights from the (k-1)-th weighted fit. Dropping this factor (the pre-2026-04-14 behavior) silently underestimates the SE by ~2x under heterogeneous weights. Verified against `lmtp::lmtp_tmle` and a 300-replicate Monte Carlo. See `vignettes/variance-theory.qmd` Section 5.4 and the inline comment block in `R/variance_if.R:variance_if_ice_one`.
- **Effect modification across methods** (`PHASE_6_INTERACTIONS.md`, all chunks done). All four methods now parse `~ L + sex + A:sex` via `parse_effect_mod()` in `R/effect_modification.R`. Point **gcomp** feeds the full formula to the outcome model directly. **IPW** (chunk 6b) expands the per-intervention MSM from `Y ~ 1` to `Y ~ 1 + modifier` via `build_ipw_msm_formula()`; the treatment is absorbed by the density-ratio weights, so only modifier main effects enter the MSM. **Matching** (chunk 6c) expands from `Y ~ A` to `Y ~ A + modifier + A:modifier` via `build_matching_msm_formula()`. Both reject only bare treatment in confounders (`~ L + A`) via `check_em_compat()`. The propensity formula strips EM terms automatically via `build_ps_formula()`. `variance_if_ipw()` and `variance_if_matching()` generalize unchanged to the expanded MSMs. **ICE** (chunk 6d) auto-expands `A:modifier` across treatment lags via `expand_em_lag_terms()` in `R/effect_modification.R`: at time_idx = k with available_lags = m, emits `lag1_A:modifier`, ..., `lag{m}_A:modifier` in addition to the current-period `A:modifier`. The expansion splits the interaction term on `:`, substitutes the treatment component with the lag prefix, and subjects each expanded term to the same all-NA column validity check as other dynamic columns. The sandwich variance engine and bootstrap require no changes (the model matrices simply grow to accommodate the additional interaction columns). Verified on a 2-period DGP (ATE|sex=0 = 5, ATE|sex=1 = 8) and a 3-period DGP (ATE|sex=0 = 8, ATE|sex=1 = 12.5) with analytical truth recovery to < 0.1.
- **Stochastic interventions are gcomp-only (Phase 12).** General stochastic interventions `g(a | L)` — where the counterfactual treatment is drawn from a user-specified distribution — are supported via the `stochastic()` constructor under `estimator = "gcomp"` only (point and ICE). The g-formula evaluates `E[Y^g]` via Monte Carlo integration: sample `M` draws from the stochastic policy per individual, predict under each, and average. Sandwich variance uses MC-averaged IFs (the IF of the MC estimator is $(1/M) \sum_m IF_m$ where each $IF_m$ is the standard gcomp IF at the m-th counterfactual draw); Channel 2's gradient input is the MC-averaged gradient, requiring no structural change to `apply_model_correction()`. IPW rejects stochastic interventions because the density-ratio engine requires either a deterministic map (for the Jacobian) or a closed-form weight (IPSI only). Matching rejects them because match weights are fixed at fit time. See `PHASE_12_STOCHASTIC.md`. Reference: Díaz & van der Laan (2012, *Biometrics* 68:541–549).
- **Outcome type coverage (Phase 13).** Supported outcome families: all `stats::` GLM families (gaussian, binomial, quasibinomial, poisson, quasipoisson, Gamma, inverse.gaussian) + any family object with `$mu.eta` and `$variance` (analytic sandwich) or any `model_fn` output (numeric fallback). **Negative binomial** already works via `model_fn = MASS::glm.nb` — the fitted object exposes `family$mu.eta`/`$variance` so the analytic sandwich path fires, but has no explicit test coverage (Phase 13). **Beta regression** for bounded-continuous outcomes works via `family = betareg::betar()` (object path in `resolve_family()`); Phase 13 extends `resolve_family()` to also accept the string `"beta"` with `betareg` in Suggests. Ordinal (`MASS::polr`) and multinomial outcomes require fundamentally different contrast/variance architecture and are deferred to Phase 13/15. See `PHASE_13_OUTCOME_TYPES.md`.
